Attention is not all you need

Why vanilla self-attention breaks down at scale, and what we actually do about it in production.

ML Transformers Systems

The original transformer paper is one of the most cited in ML history, and for good reason. But the mechanism it describes, vanilla scaled dot-product attention, has a problem that becomes obvious the moment you try to run it on anything longer than a few thousand tokens.

The cost is quadratic. For a sequence of length $n$, computing full self-attention requires $O(n^2)$ time and memory. Double the context window, quadruple the compute. This is fine for short sequences but falls apart fast.

The standard formulation

The attention mechanism takes three matrices (queries, keys, values) and computes a weighted sum. The formula from the original paper:

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

Where $d_k$ is the dimension of the key vectors. The $\sqrt{d_k}$ scaling factor keeps the dot products from getting too large, which would push the softmax into regions with tiny gradients.

A basic implementation looks like this:

python
import torch
import torch.nn.functional as F

def scaled_dot_product_attention(Q, K, V, mask=None):
    # Q, K, V: (batch, heads, seq_len, d_k)
    d_k = Q.size(-1)
    scores = torch.matmul(Q, K.transpose(-2, -1)) / d_k ** 0.5

    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))

    weights = F.softmax(scores, dim=-1)
    return torch.matmul(weights, V)

Simple enough. The problem is that scores tensor: it's of shape (batch, heads, seq_len, seq_len). With a 128k context window, that's a 128,000 x 128,000 matrix per head per batch element. You run out of memory before you run out of patience.

Where it breaks

I ran into this recently while building an inference pipeline for a retrieval system. The model needed to attend over long retrieved passages concatenated together. Here's roughly what the memory looks like:

  • At 2,048 tokens: ~16 MB per attention layer (manageable)
  • At 8,192 tokens: ~256 MB per layer (tight on consumer GPUs)
  • At 32,768 tokens: ~4 GB per layer (most GPUs can't do this)
  • At 131,072 tokens: ~64 GB per layer (not happening)

The scaling is brutal. And this is just one layer. A model with 32 layers multiplies these numbers accordingly.

The real constraint on context length is not the model architecture. It's the attention matrix. Everything else in a transformer scales linearly.

FlashAttention: the practical fix

FlashAttention doesn't change what gets computed. It changes how. Instead of materializing the full $n \times n$ attention matrix, it computes attention in tiles, keeping everything in fast SRAM and never writing the full matrix to GPU memory.

The tiling approach

The core idea: break Q, K, V into blocks and compute attention one block at a time. This requires a way to combine the softmax across blocks, which is the tricky part. The solution uses the online softmax trick from Milakov and Gimelshein (2018).

In pseudocode, the outer loop looks like this:

python
def flash_attention_forward(Q, K, V, block_size=256):
    seq_len = Q.shape[0]
    output = torch.zeros_like(Q)
    row_max = torch.full((seq_len,), float('-inf'))
    row_sum = torch.zeros(seq_len)

    for j in range(0, seq_len, block_size):
        K_block = K[j : j + block_size]
        V_block = V[j : j + block_size]

        # Score against this block
        scores = Q @ K_block.T / (Q.shape[-1] ** 0.5)

        # Online softmax: update running max and sum
        block_max = scores.max(dim=-1).values
        new_max = torch.maximum(row_max, block_max)

        # Rescale previous output and accumulate
        correction = torch.exp(row_max - new_max)
        new_sum = correction * row_sum + torch.exp(
            scores - new_max.unsqueeze(-1)
        ).sum(dim=-1)

        output = output * correction.unsqueeze(-1)
        output += torch.exp(
            scores - new_max.unsqueeze(-1)
        ) @ V_block

        row_max = new_max
        row_sum = new_sum

    return output / row_sum.unsqueeze(-1)

This is a simplified version. The real implementation is a fused CUDA kernel that operates on SRAM tiles. But the algorithmic idea is the same: you never need to hold the full attention matrix in memory.

What changes in practice

The memory complexity drops from $O(n^2)$ to $O(n)$. The compute is identical, but the wall-clock time also improves because you avoid slow HBM reads and writes. On an A100, FlashAttention-2 gets about 50-73% of theoretical peak FLOPS, compared to around 25-40% for the standard implementation.

Beyond FlashAttention

FlashAttention solves the memory problem but attention is still $O(n^2)$ in compute. For truly long contexts, there are approaches that reduce the computational complexity itself.

Sparse attention patterns

Not every token needs to attend to every other token. Local windowed attention lets each token attend to its $w$ nearest neighbors, giving $O(n \cdot w)$ cost. Models like Longformer and BigBird combine local windows with a few global tokens that attend everywhere.

Linear attention

If you can decompose the softmax kernel as $\text{softmax}(QK^T) \approx \phi(Q)\phi(K)^T$ for some feature map $\phi$, you can rearrange the computation:

$$\text{Attention}(Q, K, V) \approx \phi(Q)\left(\phi(K)^T V\right)$$

The inner product $\phi(K)^T V$ is of shape $(d, d)$ regardless of sequence length, so you compute it once in $O(n \cdot d^2)$ and then multiply each query into it. Total cost: $O(n \cdot d^2)$ instead of $O(n^2 \cdot d)$. When $n \gg d$, this is a big win.

State space models

Mamba and its successors sidestep attention entirely. Instead of computing pairwise interactions, they maintain a compressed state that gets updated token by token. The recurrence looks like:

$$h_t = \bar{A} h_{t-1} + \bar{B} x_t$$ $$y_t = C h_t$$

Where $\bar{A}$ and $\bar{B}$ are discretized versions of continuous parameters. This gives $O(n)$ sequential cost, but the trade-off is that you lose the ability to do arbitrary pairwise lookups. In practice, hybrid models (attention layers mixed with SSM layers) seem to do best.

What I actually use

For production systems I've landed on a few rules of thumb:

  1. If your context fits in 8k tokens, use FlashAttention and don't think about it further. The standard transformer with FlashAttention-2 is fast, well-supported, and battle-tested.
  2. If you need 8k-128k context, use a model that was trained with grouped query attention (GQA) and FlashAttention. GQA reduces the KV cache size, which is the actual bottleneck during inference.
  3. If you need 128k+ context, you're probably better off with a retrieval system that chunks your data and only feeds relevant pieces to the model. The quality of 200k-token context windows is still worse than good retrieval over the same data.

People keep asking for longer context windows. What they usually want is better retrieval.

Grouped query attention: a quick note

GQA is worth understanding because it addresses a different bottleneck. In standard multi-head attention, each head has its own key and value projections. During autoregressive inference, you cache all these KV pairs. With 32 heads and 128 dimensions per head, that's $32 \times 128 \times 2 = 8192$ values cached per token per layer.

GQA shares key and value heads across groups of query heads. If you have 32 query heads but only 8 KV heads, your KV cache is 4x smaller. The quality loss is negligible.

python
class GroupedQueryAttention(nn.Module):
    def __init__(self, d_model, n_heads, n_kv_heads):
        super().__init__()
        self.n_heads = n_heads
        self.n_kv_heads = n_kv_heads
        self.head_dim = d_model // n_heads
        self.n_rep = n_heads // n_kv_heads  # heads per KV group

        self.q_proj = nn.Linear(d_model, n_heads * self.head_dim)
        self.k_proj = nn.Linear(d_model, n_kv_heads * self.head_dim)
        self.v_proj = nn.Linear(d_model, n_kv_heads * self.head_dim)
        self.o_proj = nn.Linear(n_heads * self.head_dim, d_model)

    def forward(self, x):
        B, L, _ = x.shape
        Q = self.q_proj(x).view(B, L, self.n_heads, self.head_dim)
        K = self.k_proj(x).view(B, L, self.n_kv_heads, self.head_dim)
        V = self.v_proj(x).view(B, L, self.n_kv_heads, self.head_dim)

        # Repeat KV heads to match query head count
        K = K.repeat_interleave(self.n_rep, dim=2)
        V = V.repeat_interleave(self.n_rep, dim=2)

        # Standard attention from here
        return self.o_proj(attention(Q, K, V))

None of this is secret knowledge. The papers are all public, the implementations are in PyTorch and Triton. But it took me a while to build intuition for which approach to reach for in which situation. The answer is almost always: use FlashAttention, keep your context reasonable, and reach for retrieval before reaching for a longer context window.
