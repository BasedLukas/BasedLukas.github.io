VibeArena

Coding assessments for the AI era
Visit site

Overview

Traditional coding assessments are obsolete because AI writes the code now. VibeArena is a HackerRank/LeetCode replacement for the AI era. The skill that matters is whether someone can effectively use AI tools to solve real problems.

Users get dropped into a browser-based VS Code environment with AI coding tools pre-installed. They solve challenges using AI assistance, and server-side tests verify their solutions. The platform includes progress tracking, leaderboards, and user tiers.

How it works

Each challenge spins up a sandboxed Daytona workspace with a full VS Code editor, terminal, and pre-configured AI tools (Claude Code, Cursor). Candidates write and test code in real time while the platform monitors their progress. Server-side test suites validate correctness automatically.

Technical details

The frontend is built with React and TypeScript, the backend runs on FastAPI with PostgreSQL, and authentication is handled by Clerk. Sandboxed coding environments are provisioned through Daytona. The entire stack is deployed on Railway.

CPU Emulator

8-bit computer built from logic gates
Read article GitHub

Overview

This project is an 8-bit CPU emulator built from the bottom up by deriving the machine's behavior from simulated logic gates rather than hard-coding CPU operations directly. I built it after spending too much time in Turing Complete and wanting to recreate that kind of machine from scratch in Python.

The resulting CPU is already Turing complete with a very small instruction set, and I later turned it into a browser experience by bundling the Python interpreter together with the project through WASM.

How it works

The CPU has six general-purpose registers plus a special input/output register, and supports four families of instructions: immediate values, copying, arithmetic / logical operations, and evaluation-based control flow. To make it programmable, I also built a simple assembler with labels that compiles assembly-like code into the underlying binary instructions.

Technical details

The implementation starts with basic logic gates and builds upward into components such as half adders, full adders, an ALU, registers, and comparison logic. Because the design is derived from the behavior of the gates themselves, changing a single gate changes the behavior of the whole CPU.

The browser version reuses the Python implementation directly by packaging the interpreter and game code with WASM, which made it possible to run the emulator and puzzle interface entirely in the browser.

Workflow Automations

Internal tools for enterprise ops

Overview

A suite of internal automation tools built for a company running on Google Workspace. The system handled email triage, CRM updates, document processing, and infrastructure log monitoring — all powered by AI agents that could parse, classify, and act on incoming data autonomously.

What it did

Incoming emails were automatically classified using AI filters, then routed to task-specific agents. These agents could parse order details into spreadsheets, draft and send context-aware replies, process job candidate applications, and flag anomalies in system logs for auto-remediation. Rules and tasks were fully configurable through an internal dashboard.

Architecture

Built as an on-premises solution with full Google Workspace integration (Gmail, Drive, Sheets) via OAuth 2.0 service accounts. All data remained within the client's enterprise environment with zero data egress — no customer data ever left their tenant. The system was designed to meet their data residency and compliance requirements from day one.

getdelv

Custom search engine
Visit site

Overview

getDelv was a highly specialized code search engine built by scraping GitHub repositories as well as coding documentation from across the web. It combined traditional keyword search with semantic search, and it was tuned for early language models that did not have reliable internet search capabilities.

The system was made available through an API and exposed through an MCP server, so models could connect to it directly and search technical documentation in a structured way.

Technical details

We ran scrapers across GitHub, specific documentation sites such as SaaS provider docs, broader web sources, and Common Crawl to collect relevant technical material. On top of that ingestion pipeline, we used prediction models to estimate when a source had changed enough to justify a rescrape.

The search layer itself was custom built as a combination of BM25 keyword retrieval and semantic search / RAG-style retrieval. That index was then exposed through an API that could be consumed directly or through the MCP server.
